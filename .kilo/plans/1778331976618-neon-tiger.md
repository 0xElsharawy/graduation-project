# Chat Implementation Plan

## Overview

Build a basic workspace chat feature for the project management application with two core surfaces: channels and direct messages. The first version should stay focused on simple, reliable conversation features without search, notifications, advanced moderation workflows, or extra chat automation.

## Goals

- Let workspace members chat in channels and one-to-one DMs.
- Keep the initial release simple enough to build and maintain quickly.
- Support the minimum needed messaging flow for a usable chat experience.
- Give admins the only special control needed in this version: deleting or locking channels.

## Scope

### In Scope

- Workspace channels.
- Direct messages between workspace members.
- Create channel.
- Join and leave channel.
- Send messages in channels and DMs.
- Edit own messages.
- Delete own messages.
- File attachments if the existing upload pattern is easy to reuse.
- Admin-only channel delete.
- Admin-only channel lock and unlock.

### Out of Scope

- Search.
- Notifications.
- Typing indicators.
- Presence indicators.
- Read receipts.
- Message threading.
- Reaction systems beyond the minimum basic need, if they are not already required elsewhere.
- Moderation roles such as moderator or channel owner.
- Channel pins.
- Bots, integrations, and automation.

## Information Architecture

### Chat Surfaces

- Workspace chat home at `/[workspace]/chat`.
- Channel page at `/[workspace]/chat/channels/[channelId]`.
- DM page at `/[workspace]/chat/dm/[conversationId]`.

### UI Regions

- Left sidebar for channels and DMs.
- Main conversation panel for messages.
- Bottom composer for sending messages.
- Simple header for the active channel or DM.

## Data Model Plan

### Recommended Structure

Use one shared thread model for both channels and DMs so the backend and frontend stay simple.

### Core Tables

- `chat_threads`
  - `id`
  - `workspaceId`
  - `type` (`channel` or `dm`)
  - `name`
  - `visibility` (`public` or `private`) for channels
  - `createdBy`
  - `lockedAt`
  - `archivedAt`
  - `createdAt`, `updatedAt`

- `chat_thread_members`
  - `id`
  - `threadId`
  - `userId`
  - `joinedAt`

- `chat_messages`
  - `id`
  - `threadId`
  - `senderId`
  - `content`
  - `editedAt`
  - `deletedAt`
  - `createdAt`, `updatedAt`

- `chat_message_attachments`
  - `id`
  - `messageId`
  - `fileUrl`
  - `fileName`
  - `fileType`
  - `fileSize`
  - `createdAt`

### Schema Rules

- Enforce workspace scoping on every record.
- Enforce one DM thread per user pair within a workspace.
- Enforce channel membership for private channels.
- Use soft deletes for messages if possible.

### API Design

#### Channel and DM APIs

| Method | Endpoint | Description |
|---|---|---|
| GET | `/workspaces/:id/chat/threads` | List channels and DMs for the current user |
| POST | `/workspaces/:id/chat/channels` | Create a channel |
| POST | `/workspaces/:id/chat/dm` | Create or resolve a DM thread |
| PATCH | `/workspaces/:id/chat/channels/:channelId` | Update basic channel settings |
| POST | `/workspaces/:id/chat/channels/:channelId/join` | Join a channel |
| POST | `/workspaces/:id/chat/channels/:channelId/leave` | Leave a channel |
| POST | `/workspaces/:id/chat/channels/:channelId/lock` | Lock a channel as admin |
| POST | `/workspaces/:id/chat/channels/:channelId/unlock` | Unlock a channel as admin |
| DELETE | `/workspaces/:id/chat/channels/:channelId` | Delete a channel as admin |

#### Message APIs

| Method | Endpoint | Description |
|---|---|---|
| GET | `/workspaces/:id/chat/threads/:threadId/messages` | List messages in a channel or DM |
| POST | `/workspaces/:id/chat/threads/:threadId/messages` | Send a message |
| PATCH | `/workspaces/:id/chat/messages/:messageId` | Edit a message |
| DELETE | `/workspaces/:id/chat/messages/:messageId` | Delete a message |
| POST | `/workspaces/:id/chat/messages/:messageId/attachments` | Upload an attachment |

### Service Responsibilities

- Resolve workspace membership before every operation.
- Check whether the user can access a thread.
- Normalize DMs so the same pair of users maps to one DM thread.
- Prevent non-admin users from deleting or locking channels.
- Keep the channel lock state in the database and respect it when sending messages.
- Apply basic authorization for message edits and deletes.
- Emit real-time events only for message and channel updates that the UI needs.

### Authorization Rules

- Workspace members can read public channels in their workspace.
- Private channels require membership.
- Any workspace member can create or open a DM unless the app decides otherwise.
- Only admins can delete channels.
- Only admins can lock or unlock channels.
- Members can edit or delete their own messages.

- Chat sidebar.

- Message list.

### Component Checklist

#### Sidebar

- Show channels and DMs.
- Highlight the active thread.
- Show basic create-channel and new-DM actions.

#### Header

- Show the channel name or DM participant name.
- Show whether a channel is locked.
- Show admin controls for delete or lock when permitted.

## Real-Time Plan

### WebSocket Layer

- Connect when the chat page loads.
- Join the active thread room.
- Handle new message, edit message, and delete message events.
- Handle channel lock, unlock, and delete events.

### Fallback Behavior

- Use polling only if real-time sync is already part of the app’s standard pattern.
- Keep the fallback minimal and limited to the active thread.

## Dependencies to Add or Reuse

- Socket.io client if the app already uses WebSockets for live updates.
- Existing file upload patterns if attachments are included.

## Recommended Implementation Order

1. Lock the basic thread model for channels and DMs.
2. Add schema and migrations.
3. Build the channel and DM endpoints.
4. Build the basic chat pages and sidebar.
5. Implement the message composer and message list.
6. Add admin-only channel delete and lock behavior.
7. Wire up the real-time layer if needed.
8. Add attachments if they are in scope.

## Validation Plan

- Test channel creation, joining, leaving, locking, and deletion.
- Test DM creation and message sending.
- Test message edit and delete permissions.
- Test admin-only channel delete and lock rules.
- Verify the UI still works without any advanced chat features.
